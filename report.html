optofmt · optimizing idiomatic
fun f() {
    if (
        unwrapped !== rootCause &&
        unwrapped !== unwrappedCause &&
        unwrapped !is CancellationException &&

Why

A condition too long for one line must wrap to stacked operands. optofmt opens the paren and lays every operand at the same body indent — one indentation size, so the operands form a clean rectangular block. ktfmt adds a continuation indent on top of the block indent, so operand 2+ drift one step further right than operand 1, with no structural meaning a reader can attach to.

Elvis (?:) operator wrap kotlin-toolchain · PetTypeFormatter.kt:44 parity

Parity — both put `?:` at the start of the continuation line (the kotlinx-majority style).

ktfmt · greedy
fun f(): PetType {
    return findPetTypes.find { it.name == text }
        ?: throw ParseException("type not found: " + text, 0)
}
optofmt · optimizing ≡ identical
fun f(): PetType {
    return findPetTypes.find { it.name == text }
        ?: throw ParseException("type not found: " + text, 0)
}
Why

Elvis is one of the operators that reads best with the operator leading the continuation line, and both engines agree: keep the left side on the first line and start the wrapped line with ?: at one indent. (This is the opposite of the chain operators like &&, which trail.)

Annotation with arguments on its own line Kotlin project · ktfmt diff study optofmt wins

optofmt drops an argument-carrying annotation onto its own line above the declaration; ktfmt keeps it inline with `fun`.

ktfmt · greedy
@JvmName("other") fun testSomething() {}
optofmt · optimizing idiomatic
@JvmName("other")
fun testSomething() {}
Why

Kotlin convention places an annotation that carries arguments (@JvmName("other"), @Test, @Deprecated(...)) on its own line directly above the declaration; only argument-less modifier-like annotations (@PublishedApi, @JvmStatic) stay inline. ktfmt keeps every annotation glued to the declaration line, so @JvmName("other") fun testSomething() {} reads as a single run and the annotation loses the visual separation a reader expects. optofmt breaks after the argument-carrying annotation (§9).

Lambda that fits stays on one line (even with return) Kotlin project · ktfmt diff study optofmt wins

optofmt · optimizing idiomatic

Why
