ktfmt rectangle vs ergonomics

Each snippet is the same Kotlin formatted two ways: ktfmt rectangle (--kotlinlang-style) on the left, and ktfmt ergonomics (per RULES.md) on the right. A third column appears when a strictly better option exists that neither produces. Both use 4-space indent and a 100-column limit.

Amber marks any line past 100 columns. ktfmt rectangle output is real; ktfmt ergonomics output is the layout entailed by the rules.

Why

Long generic type-argument list Exposed · EntityCache.kt

A generic type-argument list is not a wrappable body, so ktfmt ergonomics keeps the type whole and breaks after `=` — landing on the same compact two-line layout as ktfmt rectangle. Parity.

ktfmt · rectangle

    internal val pendingInitializationLambdas =
        IdentityHashMap<Entity<Any>, MutableList<(Entity<Any>) -> Unit>>()
}
ktfmt · ergonomics ≡ identical

    internal val pendingInitializationLambdas =
        IdentityHashMap<Entity<Any>, MutableList<(Entity<Any>) -> Unit>>()

Why

When the declaration overflows, §3 keeps the introducer (=) attached only when the right-hand side has a wrappable body — and §4 names exactly which: call arguments, parameters, collection literals, when entries. A generic type-argument list <…> is none of those (nor a call chain, §7, nor a nested call group, §5): IdentityHashMap<…>() is a single type applied to an empty (). With no §3 body to wrap, the only break that clears the overflow is after =.
